# Monitoring Overview & Setup

This document is the **single source of truth** for all Basic and Advanced Monitoring components: Uptime, Error Tracking, Logs Aggregation, Analytics, and Self-hosted metrics.

## 1. Uptime Monitoring (UptimeRobot / Pingdom)

To ensure the application is reachable and the storage layer is operational, configure your uptime monitor to check the following endpoint:

- **URL**: `https://aurora-derm.com/api.php?resource=health`
- **Method**: `GET`
- **Expected Status**: `200 OK`
- **Expected Keyword**: `"ok": true` (Verifies storage is writable and healthy).

If the endpoint returns `503 Service Unavailable`, investigate storage/encryption immediately.

## 2. Error Tracking (Sentry)

Sentry tracks exceptions across backend and frontend.

### Configuration

- **Backend (PHP)**:
    ```bash
    AURORADERM_SENTRY_DSN="https://examplePublicKey@o0.ingest.sentry.io/0"
    ```

- **Frontend (JS)**:
    ```bash
    AURORADERM_SENTRY_DSN_PUBLIC="https://examplePublicKey@o0.ingest.sentry.io/0"
    ```

- **Environment Name**:
    ```bash
    AURORADERM_SENTRY_ENV="production"
    ```

### Evidence Verification
Sentry integrations are tracked via normalized runtime artifacts in `verification/runtime/sentry-events-last.json` generated by `npm run verify:sentry:events`. The official operational view is consumed by `node bin/prod-readiness-summary.js`.

## 3. Logs Aggregation (Papertrail)

The application uses `Monolog` to send application logs and audit events to Papertrail via UDP syslog.

### Configuration

Set the following environment variables:

```bash
PAPERTRAIL_HOST="logsN.papertrailapp.com"
PAPERTRAIL_PORT="XXXXX"
```

If Papertrail is unreachable or not configured, the application falls back safely to the standard server `error_log` (`stderr`).

## 4. Web Analytics (GA4 & Clarity)

Frontend analytics are configured via the `monitoring-config` endpoint and loaded asynchronously to prevent blocking the critical rendering path.

- **Google Analytics 4 (GA4)**: Configured via the administrative panel or `GA_MEASUREMENT_ID` environment variable. Tracks page views and conversion events.
- **Microsoft Clarity**: Configured via `CLARITY_ID`. Tracks user sessions, heatmaps, and rage clicks.

## 5. Canonical Production Monitor

Production monitor evidence follows the pattern:
- Command: `npm run monitor:prod`
- Workflow: `.github/workflows/prod-monitor.yml`
- Canonical consumer: `node bin/prod-readiness-summary.js`

This script provides a unified view of Sentry evidence, Weekly KPIs, and Production health.

## 6. Self-Hosted Grafana + Prometheus (Advanced)

For advanced local monitoring:
1. Export the same diagnostics token that the app will accept:
   ```bash
   export AURORADERM_DIAGNOSTICS_ACCESS_TOKEN=prometheus-local-token
   ```
2. Start the local PHP server on the Docker-visible port:
   ```bash
   php -S 0.0.0.0:8080 -t .
   ```
3. Start the monitoring stack:
   ```bash
   docker-compose -f docker-compose.monitoring.yml up -d
   ```
4. Access Prometheus at `http://localhost:9090` and Grafana at `http://localhost:3000` (admin/admin).

Prometheus scrapes `http://host.docker.internal:8080/api.php?resource=metrics` with
the bearer token from `AURORADERM_DIAGNOSTICS_ACCESS_TOKEN`, so the stack no
longer depends on legacy `pielarmonia` names or on unauthenticated diagnostics.

Minimum alerting wired in this repo:
- `AuroraDermQueueBacklogHigh`: queue backlog stays above `20`.
- `AuroraDermApiErrorRateHigh`: API 5xx ratio over 5 minutes exceeds `5%`.
