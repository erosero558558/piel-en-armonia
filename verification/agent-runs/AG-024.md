# AG-024 - Repair Git Sync false-failure hardening (local)

## Goal
Reducir falsos rojos del flujo `repair-git-sync` cuando la web remota esta en una revision frontend distinta pero `HEAD` local no toca assets frontend.

## Implementado
- `VERIFICAR-DESPLIEGUE.ps1`
  - Unificacion de criterio advisory para drift remoto cuando `HEAD` no cambia frontend.
  - Referencias de `index` (`script.js`, `styles.css`, `styles-deferred.css`) bajan a `WARN` en ese contexto.
  - Mismatches de hash de assets frontend bajan a `WARN` (con `AdvisoryReason=head_no_frontend_changes`) y no cuentan como fallo final.
- `.github/workflows/repair-git-sync.yml`
  - `Smoke rapido post-repair` corre con `if: always()` para diagnostico aunque falle verify
  - Issue de alerta incluye outcomes de `ssh_repair`, `verify_after_repair`, `smoke_post_repair`
  - Paso de autocierre cuando el workflow se recupera
- `SMOKE-PRODUCCION.ps1`
  - Fix portable para `pwsh` en runners Linux cuando `$env:TEMP` no existe
  - Fix portable para usar `curl` nativo en Linux/macOS y `curl.exe` en Windows

## Reproduccion / validacion local
- Re-ejecutado `VERIFICAR-DESPLIEGUE.ps1` (2026-02-25)
  - Contexto detectado: `Deploy freshness en modo advisory: HEAD no cambia assets frontend`
  - Mismatch remoto de `styles.css` y `script.js` marcado como `WARN` (advisory)
  - Resultado final: `verificacion de despliegue OK` (exit 0)
