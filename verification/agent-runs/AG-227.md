# AG-227 - Pause WhatsApp telemedicine bookings when clinical storage is not ready

## Summary

- Added a clinical-storage guard to the WhatsApp OpenClaw booking orchestrator before live telemedicine bookings proceed for `cash`, `transfer`, and `card`.
- Normalized booking failures so telemedicine flows return a consultorio fallback message when encrypted clinical storage is unavailable, including the post-checkout card finalization path.
- Exposed WhatsApp OpenClaw health metrics through `HealthController` and added diagnostics coverage for the authorized health response.
- Added integration coverage for telemedicine `cash`, `transfer`, `card`, and the card webhook manual-review behavior while the clinical storage gate is active.

## Verification

```powershell
php -l controllers/HealthController.php
php -l lib/whatsapp_openclaw/ConversationOrchestrator.php
php -l tests/Integration/HealthVisibilityTest.php
php -l tests/Integration/WhatsappOpenclawControllerTest.php
node bin/run-phpunit.js tests/Integration/HealthVisibilityTest.php
node bin/run-phpunit.js tests/Integration/WhatsappOpenclawControllerTest.php
```

## Outcome

- `6/6` integration tests passing in `tests/Integration/HealthVisibilityTest.php`.
- `12/12` integration tests passing in `tests/Integration/WhatsappOpenclawControllerTest.php`.
- Telemedicine bookings from WhatsApp now stop before creating holds, appointments, or checkout sessions when encrypted clinical storage is not ready.
- If a card checkout was opened before the gate turns on, the payment webhook now routes the case to manual review with a consultorio fallback reply instead of auto-booking telemedicine.
- Authorized health diagnostics now expose the WhatsApp OpenClaw snapshot, including bridge mode plus started/completed payment counters.
