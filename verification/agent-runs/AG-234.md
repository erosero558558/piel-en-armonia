# AG-234 - Make Windows hosting stack boot without interactive login

## Summary

- Refactored the Windows hosting bootstrap so the public stack can run in a
  boot-safe mode without depending on the interactive operator profile.
- Kept the OpenClaw bridge as a login-only component because it depends on the
  operator's local OAuth session.
- Added resilient login fallbacks via Startup shim and `HKCU\Run`.

## Changes

- Updated `scripts/ops/setup/ARRANCAR-HOSTING-WINDOWS.ps1`
    - Added `-OperatorUserProfile` so Cloudflare tunnel credentials resolve from
      the operator profile even when the public stack is launched by another
      principal.
    - Added explicit binary overrides:
      `-CaddyExePath`, `-CloudflaredExePath`, `-PhpCgiExePath`.
    - Added `-SkipBridge` so the public stack can boot without trying to start
      the operator auth bridge under the wrong user context.
    - Uses the active listener on `127.0.0.1:9000` as the source of truth for
      `php-cgi` and cleans stale duplicates instead of spawning a second FastCGI
      worker on boot/login overlap.
- Updated `scripts/ops/setup/CONFIGURAR-HOSTING-WINDOWS.ps1`
    - Resolves and persists explicit executable paths into launch commands.
    - Generates short launchers in `data/runtime/hosting/*.cmd` so Task
      Scheduler does not hit the `/TR` 261-character limit.
    - Installs `Startup` + `HKCU\Run` login fallbacks.
    - Attempts to install an `ONSTART` task running as `SYSTEM` only when the
      current PowerShell session is elevated.
    - Emits an explicit warning when the session is not elevated.
- Updated `docs/DEPLOY_HOSTING_PLAYBOOK.md`
    - Documented the dual-startup model:
      public stack on boot when elevated, bridge on operator login.
