## AG-224 - Gate telemedicine payment and appointment flows on clinical storage readiness

### Alcance

- `PaymentController` bloquea `payment_intent` de telemedicina y `transfer_proof` cuando el almacenamiento clinico no cumple cifrado.
- `AppointmentController` bloquea `appointment_store` para telemedicina bajo el mismo gate y evita hidratar `patient_cases` en `update` cuando el storage clinico no esta listo.
- `BookingService` devuelve `clinical_storage_not_ready` antes de crear reservas de telemedicina cuando el storage clinico no es compliant.
- `StorePersistence` deja de hidratar patient-flow automaticamente cuando el entorno no es compliant con cifrado.

### Checklist critica

- [x] Scope limitado a pagos, booking de telemedicina y hydration de patient-flow.
- [x] Gate clinico reutiliza `InternalConsoleReadiness` o el contrato de `storage_encryption_compliant()` segun la capa.
- [x] Pruebas de integracion cubren `payment_intent`, `transfer_proof`, `appointment_store`, `booking service` y `appointment update`.
- [x] Verificacion de sintaxis y PHPUnit ejecutados.

### Resultado

- La telemedicina ya no puede crear reservas desde `BookingService`, crear intentos de pago, registrar comprobantes ni reservar citas cuando el storage clinico sigue bloqueado por cifrado.
- Los updates administrativos de citas siguen funcionando, pero sin hidratar `patient_cases` mientras el storage clinico no este disponible.
- La hidratacion automatica desde `StorePersistence` queda alineada con el mismo gate para evitar crear artefactos clinicos en fallback no compliant.
