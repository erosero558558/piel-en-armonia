# CDX-007 - Telemedicine backend rebuild

Date: 2026-03-03
Owner: Ernesto
Executor: codex
Lane: codex_backend_ops

## Goal

Rebuild telemedicine as a first-class backend domain without changing the public frontend contract.

## Scope delivered

Implemented an internal telemedicine backend that keeps the legacy public flow on:

- `POST /api.php?resource=payment-intent`
- `POST /api.php?resource=appointments`
- `POST /api.php?resource=transfer-proof`

while routing telemedicine services through a dedicated intake domain.

## What changed

### 1. New telemedicine domain

Added `lib/telemedicine/**` with:

- `TelemedicineChannelMapper`
- `TelemedicineConsentSnapshot`
- `TelemedicineSuitabilityEvaluator`
- `TelemedicineEncounterPlanner`
- `TelemedicineRepository`
- `ClinicalMediaService`
- `TelemedicineIntakeService`
- `LegacyTelemedicineBridge`
- `TelemedicineBackfillService`

Core behaviors:

- `telefono -> phone`
- `video -> secure_video`
- suitability shadow result persisted as `fit | review_required | unsuitable`
- consent snapshot persisted with policy/legal version metadata
- encounter plan generated for remote visits
- telemedicine intakes linked to normal appointments

### 2. Storage and persistence

Extended JSON and SQLite persistence with:

- `telemedicine_intakes`
- `clinical_uploads`

Also added private clinical media storage support outside the public payment-proof flow.

### 3. Legacy bridge integration

Integrated bridge behavior into existing backend flows:

- `PaymentController` now creates or updates telemedicine draft intake in telemedicine services
- `transfer-proof` now stages legacy uploads for later classification
- `BookingService` finalizes intake, claims clinical media, evaluates suitability and decorates appointments with telemedicine metadata

### 4. Operational read models

Updated backend outputs so admin/ops can observe telemedicine state additively:

- admin data payload preserves new telemedicine collections
- calendar event descriptions include telemedicine metadata without exposing private media URLs
- staff email includes telemedicine suitability/escalation context without leaking private media
- backup health now counts telemedicine entities

### 5. Backfill and docs

Added historical backfill script:

- `bin/backfill-telemedicine-intakes.php`

Updated API docs/OpenAPI to describe the internal bridge model and additive appointment metadata.

## Key implementation notes

- No frontend files were touched.
- No new public telemedicine endpoints were introduced in this phase.
- Suitability runs in shadow mode and does not hard-block the existing booking flow yet.
- Clinical media and payment proof are now logically separated in backend processing.

## Conclusion

Telemedicine no longer depends only on `service=telefono|video`. The backend now persists telemedicine intake state, computes suitability in shadow mode, separates clinical media from payment proof, links telemedicine metadata to appointments, supports JSON/SQLite persistence, and provides a historical backfill path without breaking the current frontend contract.
