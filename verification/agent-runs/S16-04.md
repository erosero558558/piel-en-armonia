# S16-04 — Minimización de tokens de backup

## Resumen

- `verify-backup.php` ahora acepta solo `AURORADERM_BACKUP_VERIFY_TOKEN` como credencial de lectura para verificar backups.
- `AURORADERM_CRON_SECRET` y `AURORADERM_DIAGNOSTICS_ACCESS_TOKEN` pasan a devolverse como `403` explícito con códigos dedicados.
- Se retiró `X-Cron-Token` de la superficie CORS del endpoint y se actualizó la configuración/documentación en `env.example.php`, `docs/RUNBOOKS.md` y `k8s/secret.yaml.example`.
- La suite `tests/Unit/VerifyBackupEndpointTest.php` se amplió para cubrir el rechazo explícito de `CRON_SECRET` y del token de diagnósticos.

## Validación

- `php -l verify-backup.php`
- `php -l tests/Unit/VerifyBackupEndpointTest.php`
- `php vendor/bin/phpunit tests/Unit/VerifyBackupEndpointTest.php`
- smoke local:
  `curl -H "Authorization: Bearer cron-secret" http://127.0.0.1:8877/verify-backup.php` → `403`
- `git diff --check`

## Resultado

- El endpoint queda aislado con un token read-only específico y ya no reutiliza secretos con mayor alcance operativo.
