# AG-204 - Public main sync canonical head-drift diagnostics

Fecha: 2026-03-12

## Objetivo

Cerrar el bloque grande de `public_main_sync` dejando tres senales canonicas y
coherentes entre backend, orchestrator, CLI y runbooks:

- `headDrift`
- `telemetryGap`
- `failureReason`

La meta fue que `jobs verify`, `status --json`, `conflicts --json`,
`codex-check --json` y `checks.publicSync` ya no reporten un `failed`
generico, sino una clasificacion accionable del drift real.

## Cambios

- `lib/public_sync.php`
    - calcula y expone `headDrift`, `telemetryGap` y `failureReason` dentro de
      `checks.publicSync`
    - endurece `telemetryGap` para requerir ausencia real de
      `currentHead/remoteHead/dirtyPaths*`
- `tools/agent-orchestrator/domain/jobs.js`
    - normaliza `head_drift`, `telemetry_gap` y `failure_reason` desde
      `local_status_file` y `health_url`
    - infiere `failure_reason=working_tree_dirty` para payloads legacy del host
    - extiende el summary con `failure_reason`, `current_head` y `remote_head`
- `tools/agent-orchestrator/domain/diagnostics.js`
    - convierte `warn.jobs.public_main_sync_failed` en warning enriquecido con
      `reason`, `head_drift`, `telemetry_gap` y metadata estructurada
    - mantiene warnings dedicados:
        - `warn.jobs.public_main_sync_head_drift`
        - `warn.jobs.public_main_sync_telemetry_gap`
    - permite que ambos se emitan aunque la policy solo tenga habilitado
      `public_main_sync_failed`
- `tests-node/orchestrator/jobs-engine.test.js`
    - cubre `head_drift`, `telemetry_gap` y `failure_reason` en status file,
      health payload y fallback `registry_only`
- `tests-node/agent-jobs-cli.test.js`
    - valida que el snapshot del CLI exponga `failure_reason`
- `tests-node/orchestrator/diagnostics.test.js`
    - valida mensajes y metadata para `failed`, `head_drift` y `telemetry_gap`
- `tests-node/agent-orchestrator-cli.test.js`
    - agrega caso integrado de `status --json` con `failure_reason` +
      `telemetry_gap`
- `docs/AGENT_ORCHESTRATION_RUNBOOK.md`
    - documenta el triage con `failureReason`, `headDrift` y `telemetryGap`
- `docs/PUBLIC_MAIN_UPDATE_RUNBOOK.md`
    - incorpora `failureReason` al runbook operativo y a los criterios de salida

## Conclusion

La parte repo-side queda cerrada: `headDrift`, `telemetryGap` y
`failureReason` ya son contrato canonico de backend, orchestrator, CLI y
runbooks.

El warning que sigue abierto ya no es deuda local de codigo. Corresponde al
estado live actual del host y ahora queda mejor clasificado como
`failure_reason=working_tree_dirty` + `telemetry_gap=true`, lo que apunta a
runtime host desactualizado o telemetria incompleta hasta que estos cambios se
publiquen y el cron vuelva a correr.
